# 마이링크 PRD

## 1. 문서 개요

- 문서명: 마이링크 PRD(기능 정의서)
- 작성 목적: 링크트리 클론 서비스인 "마이링크"의 핵심 기능과 구현 범위를 정의한다.
- 작성 범위: 프로젝트 개요, 핵심 기능 목록, 각 기능의 상세 설명
- 제외 범위: 사용자 시나리오, 와이어프레임, 상세 UI 설계, 개발 일정
- 참고 계획: 사용자 시나리오와 와이어프레임은 이후 `docs` 폴더에 별도 문서로 작성한다.

## 2. 프로젝트 개요

### 2.1 프로젝트명

마이링크

### 2.2 프로젝트 목적

마이링크는 사용자가 자신의 여러 링크를 하나의 프로필 페이지에 모아 공유할 수 있는 링크트리 클론 서비스다. 사용자는 SNS, 블로그, 포트폴리오, 쇼핑몰, 연락처 등 여러 외부 링크를 한 페이지에 정리하고, 하나의 대표 URL만 공유하면 된다.

### 2.3 대상 사용자

- 개인 크리에이터: 유튜버, 인스타그램 운영자, 블로거, 스트리머
- 대학생 및 취업 준비생: 포트폴리오, GitHub, 이력서, 블로그 링크를 한 곳에 정리하려는 사용자
- 프리랜서: 작업물, 문의 채널, SNS를 간단히 공유하려는 사용자
- 소상공인: 예약 링크, 메뉴, 스마트스토어, 인스타그램을 한 페이지에 모으려는 사용자

### 2.4 핵심 가치

- 여러 링크를 하나의 URL로 공유할 수 있다.
- 별도 코딩 없이 개인 프로필 링크 페이지를 만들 수 있다.
- 모바일에서도 보기 좋은 간단한 랜딩 페이지를 제공한다.
- 사용자가 링크를 직접 추가, 수정, 삭제, 정렬할 수 있다.

## 3. 현재 전제

- 첫 버전은 웹 기반 서비스로 제작한다.
- 모바일 사용 비중이 높기 때문에 반응형 화면을 기본으로 한다.
- 초기 버전은 복잡한 결제, 팀 기능, 고급 분석보다 링크 관리와 공유 기능에 집중한다.
- 로그인한 사용자만 자신의 페이지를 만들고 관리할 수 있다.
- 방문자는 로그인 없이 공개 프로필 페이지를 볼 수 있다.

## 5. 기능 상세 설명

### 5.1 회원가입/로그인

사용자는 이메일과 비밀번호로 계정을 생성하고 로그인할 수 있어야 한다.

#### 요구사항

- 사용자는 이메일, 비밀번호를 입력해 회원가입할 수 있다.
- 사용자는 가입한 이메일과 비밀번호로 로그인할 수 있다.
- 로그인한 사용자만 관리자 화면에 접근할 수 있다.
- 로그아웃 기능을 제공한다.
- 잘못된 이메일 또는 비밀번호 입력 시 명확한 오류 메시지를 표시한다.

#### 예외 상황

- 이미 가입된 이메일로 회원가입을 시도하면 중복 안내를 표시한다.
- 비밀번호가 최소 조건을 만족하지 않으면 가입을 막는다.
- 로그인 세션이 만료되면 다시 로그인하도록 안내한다.

### 5.2 프로필 페이지 생성

회원가입 후 사용자는 자신의 공개 프로필 페이지를 가질 수 있어야 한다.

#### 요구사항

- 사용자마다 하나의 기본 프로필 페이지를 생성한다.
- 공개 프로필 페이지는 고유 URL을 가진다.
- 방문자는 로그인 없이 공개 프로필 페이지에 접근할 수 있다.
- 프로필 페이지에는 사용자 이름, 소개글, 프로필 이미지, 링크 목록이 표시된다.

#### 기본 URL 예시

```text
https://mylink.example.com/u/username
```

### 5.3 프로필 정보 관리

사용자는 자신의 프로필 페이지에 표시될 기본 정보를 관리할 수 있어야 한다.

#### 관리 항목

- 표시 이름
- 소개글
- 프로필 이미지
- 대표 배경색 또는 테마

#### 요구사항

- 표시 이름은 필수 입력값이다.
- 소개글은 선택 입력값이다.
- 프로필 이미지는 선택 입력값이다.
- 저장 후 공개 페이지에 즉시 반영되어야 한다.

### 5.4 링크 추가

사용자는 자신의 프로필 페이지에 표시할 외부 링크를 추가할 수 있어야 한다.

#### 입력 항목

- 링크 제목
- 링크 URL
- 링크 설명(선택)

#### 요구사항

- 링크 제목과 URL은 필수 입력값이다.
- URL은 올바른 형식인지 검증해야 한다.
- 새 링크는 기본적으로 목록의 마지막에 추가된다.
- 링크 추가 후 관리자 화면의 링크 목록에 바로 표시된다.

#### 예시

- 제목: GitHub
- URL: `https://github.com/username`
- 설명: 개발 프로젝트 모음

### 5.5 링크 목록 조회

사용자는 자신이 등록한 링크들을 관리자 화면에서 확인할 수 있어야 한다.

#### 요구사항

- 등록된 링크 제목, URL, 상태를 목록으로 보여준다.
- 링크가 없는 경우 빈 상태 화면을 표시한다.
- 각 링크 항목에는 수정, 삭제, 순서 변경 기능으로 이어지는 조작 요소가 있어야 한다.

### 5.6 링크 수정

사용자는 기존 링크의 제목, URL, 설명을 수정할 수 있어야 한다.

#### 요구사항

- 수정 화면 또는 인라인 편집 UI를 제공한다.
- URL 수정 시 형식 검증을 다시 수행한다.
- 저장 후 공개 페이지에 즉시 반영되어야 한다.
- 취소 시 기존 내용이 유지되어야 한다.

### 5.7 링크 삭제

사용자는 더 이상 필요 없는 링크를 삭제할 수 있어야 한다.

#### 요구사항

- 삭제 버튼을 제공한다.
- 실수 방지를 위해 삭제 확인 절차를 제공한다.
- 삭제된 링크는 공개 페이지에서 더 이상 표시되지 않는다.

### 5.8 링크 정렬

사용자는 공개 페이지에 표시되는 링크 순서를 변경할 수 있어야 한다.

#### 요구사항

- 링크 순서를 위/아래 이동 또는 드래그 앤 드롭 방식으로 변경할 수 있다.
- 변경된 순서는 저장되어 공개 페이지에 반영된다.
- 모바일 환경에서도 정렬 기능을 사용할 수 있어야 한다.

### 5.9 공개 페이지 조회

방문자는 사용자의 공개 프로필 페이지를 볼 수 있어야 한다.

#### 요구사항

- 방문자는 로그인 없이 페이지를 볼 수 있다.
- 페이지에는 프로필 정보와 활성화된 링크 목록이 표시된다.
- 링크 클릭 시 새 탭 또는 현재 탭에서 외부 페이지로 이동한다.
- 없는 사용자 URL에 접근하면 404 또는 안내 페이지를 표시한다.

### 5.10 반응형 UI

서비스는 모바일, 태블릿, 데스크탑에서 자연스럽게 동작해야 한다.

#### 요구사항

- 모바일에서는 세로형 단일 컬럼 레이아웃을 사용한다.
- 태블릿에서는 콘텐츠 폭을 제한해 가독성을 유지한다.
- 데스크탑에서는 중앙 정렬된 카드 또는 넓은 대시보드 레이아웃을 사용한다.
- 버튼과 입력 요소는 터치하기 충분한 크기를 가져야 한다.

### 5.11 테마 커스터마이징

사용자는 자신의 공개 페이지 스타일을 간단히 변경할 수 있다.

#### 요구사항

- 배경색을 선택할 수 있다.
- 링크 버튼 스타일을 선택할 수 있다.
- 텍스트 색상은 배경과 대비가 충분해야 한다.
- 초기 버전에서는 미리 정의된 테마를 제공하는 방식이 적합하다.

#### 상태

선택 기능

### 5.12 링크 활성/비활성

사용자는 링크를 삭제하지 않고 임시로 숨길 수 있어야 한다.

#### 요구사항

- 각 링크마다 활성/비활성 상태를 설정할 수 있다.
- 비활성 링크는 관리자 화면에서는 보이지만 공개 페이지에서는 보이지 않는다.
- 다시 활성화하면 기존 순서에 맞게 공개 페이지에 표시된다.

#### 상태

선택 기능

### 5.13 클릭 통계

사용자는 각 링크가 얼마나 클릭되었는지 확인할 수 있다.

#### 요구사항

- 링크별 총 클릭 수를 표시한다.
- 기간별 클릭 수는 추후 확장 기능으로 고려한다.
- 방문자 개인을 식별하지 않는 방식으로 통계를 수집한다.

#### 상태

선택 기능

## 6. 관리자 화면 구성 방향

관리자 화면은 사용자가 자신의 링크 페이지를 빠르게 수정할 수 있도록 단순해야 한다.

### 주요 영역

- 프로필 정보 편집 영역
- 링크 목록 관리 영역
- 링크 추가 버튼
- 공개 페이지 미리보기 또는 보기 버튼

### 설계 원칙

- 가장 자주 쓰는 기능인 링크 추가와 순서 변경을 쉽게 접근할 수 있게 한다.
- 저장 결과가 공개 페이지에 반영되는지 명확하게 보여준다.
- 복잡한 설정은 초기 화면에 과하게 노출하지 않는다.

## 7. 공개 페이지 구성 방향

공개 페이지는 방문자가 사용자의 링크를 빠르게 이해하고 클릭할 수 있어야 한다.

### 주요 영역

- 프로필 이미지
- 이름
- 소개글
- 링크 버튼 목록
- SNS 아이콘 영역(선택)

### 설계 원칙

- 모바일 화면에서 링크 버튼이 잘 보여야 한다.
- 버튼 간 간격을 충분히 둔다.
- 사용자 개성을 표현할 수 있는 테마를 제공한다.
- 링크 클릭 흐름을 방해하는 요소를 최소화한다.

## 9. 비기능 요구사항

### 성능

- 공개 프로필 페이지는 빠르게 로드되어야 한다.
- 모바일 네트워크 환경에서도 링크 목록이 지연 없이 표시되어야 한다.

### 보안

- 비밀번호는 평문으로 저장하지 않는다.
- 사용자 입력값은 검증하고 필요한 경우 이스케이프 처리한다.
- 다른 사용자의 프로필이나 링크를 수정할 수 없도록 권한을 검증한다.

### 접근성

- 버튼과 링크는 키보드로 접근 가능해야 한다.
- 색상 대비를 충분히 확보한다.
- 이미지에는 대체 텍스트를 제공한다.

### 유지보수

- 사용자 시나리오, 와이어프레임, API 설계 문서는 별도 파일로 분리한다.
- 기능 추가 시 필수 기능과 선택 기능의 범위를 명확히 관리한다.

## 10. MVP 범위 제안

첫 버전에서는 아래 기능까지를 MVP로 제안한다.

- 회원가입/로그인
- 프로필 정보 관리
- 링크 추가
- 링크 수정
- 링크 삭제
- 링크 정렬
- 공개 페이지 조회
- 반응형 UI

아래 기능은 MVP 이후 개선 단계에서 추가하는 것을 제안한다.

## 11. 개선 제안

### 11.1 공개 URL 정책을 먼저 정하기

링크트리류 서비스에서 공개 URL은 핵심 경험이다. 초기에 `username` 중복 정책, 허용 문자, 변경 가능 여부를 정해두는 것이 좋다.

제안 규칙:

- 영문 소문자, 숫자, 하이픈만 허용
- 3자 이상 30자 이하
- 이미 사용 중인 username은 등록 불가
- 변경 가능하되 너무 잦은 변경은 제한

### 11.2 미리보기 기능 추가

관리자가 링크를 수정할 때 공개 페이지가 어떻게 보이는지 바로 확인할 수 있으면 사용성이 좋아진다.

제안 방식:

- MVP에서는 "공개 페이지 보기" 버튼 제공
- 이후에는 관리자 화면 오른쪽에 실시간 미리보기 추가

### 11.3 링크 안전성 안내

사용자가 잘못된 링크를 등록하거나 방문자가 위험한 외부 링크로 이동할 수 있으므로 기본적인 URL 검증이 필요하다.

제안 방식:

- `http://` 또는 `https://` URL만 허용
- 비정상 URL 저장 방지
- 클릭 시 외부 사이트로 이동한다는 기본 브라우저 동작 유지

## 13. 오픈 질문
